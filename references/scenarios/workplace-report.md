# Scenario: 职场汇报 (Workplace Reports)

Covers: 周报/月报 (weekly/monthly updates), 项目复盘 (project retrospectives), 述职 (performance/promotion reviews). These share an audience (a manager or a room of peers, not an external client) and a common failure mode: burying the one thing that matters under a complete chronological account of everything that happened.

## Structure by sub-type

**周报/月报 (regular status updates)**
1. 结论先行：本周期最重要的一件事（好消息或坏消息，不要埋在中间）
2. 进展 vs 目标：用同一个对比维度贯穿全篇（不要每页换一种衡量方式）
3. 风险/问题：明确写出需要谁来帮忙解决，不要只是罗列困难
4. 下周期计划：具体到可验收的动作，不是模糊的方向性描述

Keep it short — a status update that takes 15 minutes to present has usually stopped being a status update. Default to fewer, denser slides over more, thinner ones.

**项目复盘 (retrospectives)**
1. 结果先说清楚：达成/未达成目标，一句话，不铺垫
2. 时间线只在需要解释"为什么"时使用，不要把复盘做成流水账
3. 每个问题配一个"下次怎么做不同"，没有对应改进动作的问题不值得单独列一页
4. 避免只讲团队做得好的部分——复盘的价值在于诚实

**述职 (performance/promotion review)**
1. 用"结果 + 你的具体贡献"结构每一段经历，不要只列职责范围
2. 数据化影响：能用数字说清楚的，不要用形容词代替（"显著提升"不如"提升23%"）
3. 讲清楚困难和你的应对，比只讲顺利的部分更有说服力
4. 结尾明确未来方向的诉求，不要让评审方猜你想要什么

## What reviewers in this context actually penalize

Based on repeated real-world feedback patterns: too many slides for the time allotted, a status update that reads as a justification for effort rather than a report of outcomes, and charts/data that don't map cleanly to the stated goal for the period. When in doubt, cut toward "what does my manager need to know and decide," not "everything I did."

## Design conventions

Keep visual style restrained relative to external-facing decks — internal audiences read status updates for information, not persuasion polish. A heavily designed weekly report can read as effort misallocated to the wrong thing. Save visual investment for decks where persuasion is actually the goal (see `business-proposal.md`).
